### 数据库表结构设计的一点思考（没有技术）

#### 怎么设计？

```
上午面试了一个项目型公司的人，我随口问了下他数据设计的原则，后来在他回答的过程中，
我有了点思考。

项目类型的系统，一般没有很多应用和很长的业务链条。
所以上下游系统不会很多。
表结构一般是做加法，将数据更过的吃进来。
同时表结构是业务字段的全身镜像。一般完美复制。简单粗暴。
表结构设计一般是在性能上考量，join的数量和表宽度的取舍。

而互联网类的系统，soa或者微服务，他们数据是流动的，是散落在各个系统中。
所以表结构设计更多聚焦在自身的核心业务，强依赖字段。做减法。
当然，不排除有些系统处于性能的考量，做加法。
但是整个方向是强调系统边界，聚焦自身的领域。
所以表结构设计是业务的骨干。

```
